## 3. S8F4 / S6F11 打印结果问题

用户截图曾显示：

```text
S8F4:
  ResultCode = 1
  Description = "The operation has timed out."

S6F11:
  ResultCode = 0
  Description = "????"
```

判断：

- 这不是 ShelfId 配置问题。
- 这是 ERACK Server 等 Unit 打印响应超时导致的结果不一致。
- 原逻辑 `ERackServer:RequestTimeoutMilliseconds=10000`，真实 Zebra 打印可能超过 10 秒。
- Server 先超时回复 S8F4 失败；Unit 端后面又完成打印并主动发 S6F11 成功。

已做的代码修正：

- 默认 `ERackServer:RequestTimeoutMilliseconds` 从 `10000` 改为 `60000`
- 以下 profile 脚本也写入 `60000`：
  - `set-profile-real-both-hardware.cmd`
  - `set-profile-real-server.cmd`
  - `set-profile-remote-both-sim.cmd`
- `ERackTcpUnitRouter` 中：
  - 读/写路由超时不再把 `.NET` 英文异常原样返回
  - 打印路由超时按打印协议映射为 `PrintProtocolResult.PrinterOffline`
  - unknown shelf 描述改为中文：`货架未在线：{ShelfId}`

尚未现场验证：

- 真实 Zebra 打印是否能在 60 秒内稳定返回 S8F4 成功。
- 如果 Zebra SDK send 本身长时间阻塞超过 60 秒，仍可能超时；需看新日志。

## 4. SECS 中文描述显示为 `????` 的问题

截图里 S6F11 的 `Description` 显示 `????`，原因很可能是：

- 代码用 SECS `A(...)` 构造 Description。
- SECS `A` 是 ASCII 字段，中文不一定能显示。
- `secs4net` 反射确认存在：
  - `A(string)`
  - `J(string)`
  - `SecsFormat.JIS8`

下一轮必须决策：

- 如果协议/客户 Host 支持 JIS8，则把 Description 字段从 `A(description)` 改为 `J(description)`，至少包括：
  - S8F4 Description
  - S10F12 Description
  - S6F11 RFID write Description
  - S6F11 print Description
- 如果客户 Host 只接受 ASCII `A`，则不要在 SECS Description 放中文；用 `ResultCode` 为准，Description 改成 ASCII 枚举，例如 `PRINT_SUCCESS`、`PRINTER_OFFLINE`，GUI/日志再显示中文。

不要再直接承诺 `A="打印成功"` 一定能在模拟器显示中文。当前证据显示会变成问号。
