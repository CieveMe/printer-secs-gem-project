# 硬件串口代码分析

来源目录：`D:\Desktop\打印机项目\ERack_Test_WinForm_20260602\ERack_Test_WinForm`

## 结论

这份代码是一个 .NET Framework 4.8 WinForms 测试工具，不是可直接引用的 DLL。核心串口协议在 `Hostif.cs`，界面按钮调用在 `Form1.cs`，串口枚举在 `my_serial_port.cs`。

它对本项目有用：可以把里面的串口收发和数据解析逻辑抽出来，改成当前 EQ 程序里的 `IHardwareGateway` 真实实现。

## 串口参数

- 串口号：从界面下拉框选择。
- 波特率：代码固定为 `57600`。
- 数据位、校验位、停止位：代码没有显式设置，按 .NET `SerialPort` 默认值处理，通常是 `8N1`。

## 数据帧格式

发送帧由 `Hostif.send_packet` 组包：

```text
AA 55
长度高字节 长度低字节
命令字
00 00
设备ID
数据区
校验高字节 校验低字节
```

长度字段写的是 `数据区长度 + 8`。校验是从帧头开始到数据区结束的简单字节求和，最后按高字节、低字节追加。

接收解析由 `Hostif.formatData` 完成：

- 校验帧头 `AA 55`。
- 按长度等待完整帧。
- 用同样的求和方式校验。
- 返回一个简化后的 `data_buff`，其中包含命令字、设备 ID 和业务数据。

## 已看到的硬件能力

- 读 RFID：`InventoryRespWaitResp`、`readInventoryTagWaitResp`。
- 设置显示内容：`setDisplayAllWaitResp`。
- 设置 LED：`setLedStateWaitResp`。
- 读取传感器状态：`getSensorStateWaitResp`。
- 读取 ST8400 显示内容：`ST8400ReadDisplayWaitResp`。
- 读取/写入设备参数：`ReadParaWaitResp`、`WriteParaWaitResp`。
- 重启设备：`RestartWaitResp`。

## RFID 读取逻辑

界面上的 RFID 测试按钮会调用 `InventoryRespWaitResp`，入参里传 `4`。成功返回时，业务数据长度按 32 字节处理。

`Form1.readerTagDataHandler` 的处理方式：

- 把 32 字节分成 4 组，每组 8 字节。
- 每组内部倒序。
- 4 组累加值都为 0 时，认为未读到标签。
- 否则按 ASCII 转成字符串。

## 显示内容写入逻辑

界面上的“设置显示”按钮会把输入框内容转 ASCII，然后追加：

```text
0D 0A
7F 7F 7F 7F 7F 7F 7F 7F 7F 7F 7F 7F 7F 7F 7F 7F
```

再调用 `setDisplayAllWaitResp` 发送给设备。

## 接到当前项目的方式

建议新增一个串口适配层，实现当前项目里的 `IHardwareGateway`：

- `QueryShelfStatusAsync`：优先映射到读 RFID 或读传感器。
- `WriteTagAsync`：需要先确认“写 tag”在现场含义。

目前源码里没有明显看到“把数据写入 RFID 芯片存储区”的命令。如果这里的“写 tag”实际指写电子货架显示内容，那么现有 `setDisplayAllWaitResp` 可以复用；如果实际指写 RFID 卡内数据，则这份代码可能还缺少对应命令。

## 需要注意的代码点

- 现有代码是 WinForms 写法，里面有静态串口、`MessageBox`、界面控件依赖，不能原样塞进服务程序。
- `blockReceiveData` 里对 RFID 返回长度的判断写成了 `data_buff.Length == 36 && data_buff.Length == 5`，这个条件不可能成立。界面能用多半是因为最后仍然返回了 `data_buff`，但真实适配层里应改成明确的成功/失败判断。
- 当前适配层应把串口号、波特率、设备 ID、等待时间都做成配置项。
