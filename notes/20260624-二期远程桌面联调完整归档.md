## 2. 项目一句话

- 同一个进程内跑 SECS/GEM 后台。
- 同一个进程内跑 RFID/ERack 串口。
- 同一个进程内跑 Zebra 打印。
- Host 指令和 GUI 按钮共用同一个硬件 gateway。

## 4. 当前二期业务全貌

一个业务单元：
- 一个 MCU/控制器连接显示屏、RFID、传感器。
- 电脑通过串口连接 MCU。
- 电脑连接 Zebra 打印机。
- MES 通过 SECS/GEM 连接电脑。

一个电脑可以连接多个 MCU：
- 通过 LOCATION ID 定位具体 MCU。
- Location ID 与 MCU 是一对一关系。
- Location ID 仍由 Unit 内部处理。

一个电脑可以连接多个打印机：
- GUI 上可选择 Zebra 打印机。
- 第一次选择后写入 App.config 的 Printer:ZebraPrinterAddress。
- 客户说“重装软件之前都不会变”，即默认打印机地址保存后长期固定。

多业务单元时：
- MES 不再直接连接每一台业务电脑。
- MES 连接 ERACK Server。
- ERACK Server 通过 ShelfId 找到对应 ERACK Unit。
- Unit 内部再通过 LocationId 找到具体 MCU。

## 5. Server / Client / Unit 概念

ERACK Server：
- 面向 MES/SECS Simulator。
- 运行 SECS/GEM。
- 监听 Unit Client 注册。
- 按 ShelfId 把 S5F11/S10F11/S8F3 转发给对应 Unit。
- 不直接碰本机 COM/Zebra，除非本机同时也是一个 Unit。

ERACK Unit / Client：
- 主动连接 ERACK Server。
- 注册自己的 ShelfId、UnitId、Locations。
- 收到 Server 转发请求后，操作本机 MCU/RFID/显示屏/打印机。
- 把本机事件 ShelfStateChanged/RfidWriteEvent/PrintEvent 发回 Server。

Runtime:Mode 三种：
- Unit：本机业务单元。默认可直连 MES；如果 ERackClient:Enabled=true，则作为客户端连接外部 Server，并且不启动本机 SECS，避免端口冲突。
- Server：只做 ERACK Server。启动 SECS/GEM 和 TCP Server，不打开本机 COM，不跑本机 Unit Client。
- Both：同一台电脑既做 Server，又注册一个本机 Unit Client。单机联调和服务器电脑本身也带一个业务单元时使用。

关键结论：
- 一台电脑既有服务端又有客户端时，只启动一次 PrinterSecsGem.Eq.exe，配置 Runtime:Mode=Both。
- 一台电脑只做服务端也支持，只启动一次 PrinterSecsGem.Eq.exe，配置 Runtime:Mode=Server。
- 两台电脑阶段，服务器电脑启动一次 exe，客户端电脑也启动一次 exe。

## 6. ERACK Server/Unit 自定协议

客户还没有提供 Server 与多个 Unit 之间的正式协议。

- Unit 主动连接 Server。
- 请求超时 10000ms。
- Unit 断线后 3000ms 重连。
- 心跳 5000ms。

JSON Lines 外壳字段：
- MessageId
- MessageType
- ShelfId
- Payload

Unit 首包：

- MES 发 S5F11/S10F11/S8F3 时，Server 按 ShelfId 转发到对应 Unit。
- 找不到 ShelfId 或 Unit 不在线时，返回失败码 6，不崩溃。

当前支持转发请求：
- ReadShelfStatus
- WriteRfid
- Print

当前支持 Unit 主动事件：
- ShelfStateChanged，Server 转成 S6F21。
- RfidWriteEvent，Server 转成 S6F11。
- PrintEvent，Server 转成 S6F11。

- S1F3 -> S1F4：设备状态。
- S5F11 -> S5F12：读 RFID / 查询格口状态。
- S10F11 -> S10F12：写 RFID tag。
- S8F3 -> S8F4：打印。

已支持主动上报：
- S6F21：格口状态变化，上架/下架/有无 RFID。
- S6F11：RFID 写入结果事件。
- S6F11：打印结果事件。

暂不实现：

原因：
客户 PDF 目前只看到 ALID 表，没有完整 S5F1 消息体格式。ALID 包括 AL2001 RFID 读取失败、AL2002 打印机离线、AL2003 RFID 设置失败、AL2004 格口号不存在。没有完整格式前，不默认乱发 S5F1。

## 8. S6F21 和 S6F11 理解

S6F21：
- 用于格口状态变化主动上报。
- 字段按 PDF：ShelfId、LocationId、RFID ID、状态、Unix 秒时间戳。
- 状态 1 表示上架/有货。
- 状态 0 表示下架/无货。
- 有货但读不到 RFID 时，RFID ID 字段为空字符串，状态仍是 1。
- 无货时，RFID ID 字段为空字符串，状态是 0。

S6F11：
- 通用事件上报。
- 当前用来上报 RFID 写入结果和打印结果。
- 写 RFID 成功/失败都可上报自定义 report。
- 自定义 report 字段包含 ShelfId、LocationId、RFID 值、ResultCode、Description、Timestamp。
- S10F12 仍然是 Host 写 RFID 命令的即时回复，S6F11 是额外主动事件，不替代 S10F12。

当前默认 CEID/RPTID：
- RfidWriteCeid=2002
- RfidWriteRptid=2002
- PrintCompletedCeid=2003
- PrintFailedCeid=2004
- PrintRptid=2003

这些编号如果客户正式表确认变化，再改 App.config。

## 9. 传感器和显示屏逻辑

客户已确认：
- MCU 不会在状态变化时主动向电脑串口推送数据。
- 只能电脑程序去轮询。
- 客户推荐轮询周期 500ms。

传感器读取：
- 使用命令 0x05。
- 取 data 段第二个字节，即 SensorPayloadIndex=1。
- sensorState == CheckLevel 表示有货。
- sensorState != CheckLevel 表示无货。
- CheckLevel 默认 0。
- CheckLevel 必须做成 App.config 配置项，因为现场有 3/4 种传感器，有的是判高，有的是判低。

有货：
- 读 RFID。
- 读到 RFID：显示 RFID 到屏幕，并发 S6F21，状态 1，RFID 字段为 tag。
- 读不到 RFID：显示 NO ID 后加失败码，默认 NO ID  2001，并发 S6F21，状态 1，RFID 字段为空字符串。

无货：
- 清空显示屏。
- 客户确认清空显示屏是 0x14，data 段 0 字节。
- 发 S6F21，状态 0，RFID 字段为空字符串。

当前配置默认：
- ERackSensorDisplay:Enabled=false
- PollIntervalMilliseconds=500
- SensorCommand=5
- SensorPayloadIndex=1
- CheckLevel=0
- NoIdDisplayText="NO ID  "
- NoIdFailureCode=2001

## 10. RFID 写入规则

旧规则：
- tag 长度必须是 8/16/24/32 个 ASCII 字符。
- 原因是 RFID 底层按 8 字节页写，旧程序把业务输入也限制为完整页。

二期规则：
- 支持 1-32 个 ASCII 字符。
- 整页长度 8/16/24/32 直接写。
- 非 8 倍数长度先读旧 tag，再覆盖前 N 字节，保留尾部旧字节，按向上取整后的页长度写回。
- 例如 9 字节会写 16 字节，第 9 字节落到第二页第一个字节，后 7 字节保留旧内容。
- 如果非整页写入前读旧 tag 失败，则写入失败，不发送写命令。
- 短写不清尾部，读 32 字节时后面可能带旧数据，这是当前确认后的行为，不是 bug。

如果现场还提示 “tag length must be 8/16/24/32 characters”：
- 说明现场 exe 还是旧写 tag 校验规则，或客户在用旧 ERack 测试程序。
- 不是硬件问题。

## 11. 错误码和等待时间沟通口径

code=0：成功。
code=2：参数校验失败，例如旧版本 tag 长度不是 8/16/24/32，或 tag 为空。通常不会发硬件命令。
code=3：内容为空或 ASCII 不合法等输入问题。
code=4/5：打印机发现或发送等打印相关错误。
code=6：ShelfId/LocationId/Unit 路由找不到或 Unit 不在线。
code=7：硬件/串口执行异常通用码，例如 COM 打不开、RFID 响应超时、返回数据格式异常。code=7 不是 SECS/GEM 协议层错误，不回 S9Fx，而是在业务回复中返回 ResultCode=7。

写 tag 等待时间：
- 先等 200ms。
- 最多 20 次读。
- 每次 ReadTimeout=200ms。
- 总体约 4200ms，即约 4.2 秒。

读 RFID 等待时间：
- 先等 50ms。
- 最多 20 次读。
- 每次 ReadTimeout=50ms。
- 总体约 1050ms，即约 1 秒。

客户曾问 “code=7 对应 secsgem 回复什么”：
- S10F11 写 tag 失败时回 S10F12，ResultCode=7，Description 带具体异常。
- S5F11 读失败时回 S5F12，ResultCode=7，Locations 为空；当前 S5F12 没 Description 字段，详情看日志。
- 这不是 SECS 协议层错误，不发 S9Fx。

## 17. 两台电脑真实业务部署思路

客户后面正式业务一开始可能就是两台电脑：

电脑 A：
- 既做 ERACK Server，又做一个本机 Unit。
- Runtime:Mode=Both。
- MES/SECS Simulator 只连电脑 A。
- ERackServer:ListenIp 建议正式网络填 0.0.0.0 或电脑 A 的局域网 IP。
- ERackClient:ServerHost 如果本机注册自己，可填 127.0.0.1。

电脑 B：
- 只做 Unit Client。
- Runtime:Mode=Unit。
- ERackClient:Enabled=true。
- ERackClient:ServerHost 填电脑 A 的 IP。
- ERackClient:ShelfId 填电脑 B 对应货架号，例如 SHELF002。
- 不启动本机 SECS。

网络：
- 电脑 A 防火墙放行 7801。
- MES/SECS Simulator 连接电脑 A 的 SECS 端口，默认 5000。
- 电脑 B 主动连电脑 A 的 7801。

无硬件先测：

- 两台都可以 Printer:RealPrintEnabled=false。
- 两台都可以 ERackSimulation:Enabled=true。

- Unit 电脑改 ERackHardware:Enabled=true。
